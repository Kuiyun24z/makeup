# 阶段 7 计划：Agent 咨询师模式（开放式多轮咨询）

> 2026-06-11 制定。前提：阶段 1–6 主链路已完成（实时分析→规则引擎→流式建议→语音/唤醒→四品类试妆）；6C 照片增强与商品库联动按用户决定暂缓。本阶段是总计划第 4 层的落地：让真正开放式的请求（多轮、跨工具、顺序不确定）由云端模型自主编排，简单请求继续走现有直通路径，不付 agent 延迟税。

## 0. 定位与触发（继承总计划，再次明确）

- Agent 只服务这类请求："下周参加婚礼，帮我挑一支显气色的口红先试试""对比一下豆沙和枫叶红哪个适合我""给我配一整套通勤妆并解释为什么"。
- 一键建议、单句指令、普通提问**不进 agent**：现有 5A 指令 → 规则引擎 → 流式解读三层路径保持原样。
- 实时链路（分析/渲染）依旧不依赖任何模型。

## 1. 架构

```text
浏览器统一入口
  ├─ 指令（5A parser 命中）────────────► 本地直接换妆
  ├─ 普通问题（路由判定：简单）───────► /api/advice/stream（现状不动）
  └─ 开放式请求（路由判定：复杂）─────► /api/agent/stream（新增）
                                            │
                                  server.py Agent Loop（云模型 + 工具）
                                            │
                  ┌─────────────┬───────────┼──────────────┬─────────────┐
            get_face_state  recommend   search_cosmetics  apply_makeup  adjust_makeup
            （请求附带的     （包装       （查询           （返回 action  （同左）
              FaceState，     rules.py）   cosmetics.json）  事件，由前端
              不重新分析）                                   执行换妆）
```

关键设计：**信息类工具在服务端执行，改变画面的工具变成 action 事件流回前端执行**——agent 不直接碰渲染，前端收到 `{type:"action", apply:"lip-maple", intensity:0.6}` 后走现有 `applyCatalogItem`，与手动/语音路径完全同构。

## 2. 模型与编排层选型（需要你拍板）

| 方案 | 优点 | 风险/成本 |
|---|---|---|
| OpenHarness 编排 + 云模型 | 项目初衷，插件/MCP 生态 | 接入摩擦未知（此前 CLI tools 兼容问题）；预算 1 次会话验证 |
| server.py 内置轻量 tool-loop（直连云 API） | ~150 行可控代码，无额外依赖 | 放弃 OpenHarness 生态（可后补） |

**模型候选（重要：考虑你的网络环境）：**

- 首选 **DashScope 通义 qwen-plus / qwen-max**：国内直连、tool use 支持成熟、按量付费便宜；
- 备选 Claude Sonnet / GPT 系列：tool 调用质量最好，但 API 需要可用的网络通道。

决策默认值：先用 1 次会话验证 OpenHarness + DashScope；摩擦超预算即切内置 tool-loop。**需要你提供：模型选择确认 + API key（写入本地 `web_demo/.agent_config.json`，已 gitignore，绝不入库）。**

## 3. 工具协议（服务端 schema）

- `get_face_state()`：返回请求附带的最新 FaceState（颜色/姿态/特征摘要）。**注意：发送给云端的只有结构化 JSON，任何图像帧都不出本地**——这是硬性隐私约束，写入 prompt 与代码双重保证。
- `recommend_makeup()`：包装 `rules.generate_draft`，返回结构化草稿。
- `search_cosmetics(query?, category?, finish?)`：在 cosmetics.json（36 条）内检索，返回候选列表。
- `apply_makeup(itemId, intensity?)` / `adjust_makeup(target, delta)` / `clear_makeup()`：产生 action 事件，前端执行后回传执行结果（成功/当前妆容状态），供 agent 下一轮决策。
- 工具结果与对话历史保留在会话内存中，支持多轮（"换个淡一点的"能理解指代）。

## 4. 路由（简单 vs 开放式）

- 第一版用启发式：含预算/对比/挑选/整套/解释类关键词，或问题长度超阈值，或显式前缀"帮我参谋"→ agent；其余 → 现有路径。
- UI 提供手动入口兜底："深度咨询"开关（或长按生成建议），路由误判时用户可强制。
- 路由函数独立纯函数 + 单测；误判成本低（走错路径只是体验差，不出错）。

## 5. 前端改动

- 顾问手记卡升级为**会话态**：agent 模式下保留多轮对话气泡（用户/顾问/工具动作三种），新会话按钮清空。
- NDJSON 事件扩展：`status`（"正在查色库…"）、`action`（执行换妆并显示"已为你试上枫叶红"）、`delta`/`done`/`error` 沿用阶段 3 协议。
- action 执行后自动回传结果到会话（继续 agent 循环）。
- 云端不可用/超时：降级提示并回落到普通建议路径，不阻塞。

## 6. 实施切分

```text
7A 路由 + /api/agent/stream 骨架 + 工具协议 + mock 模型集成测试   —— 不需要 API key 即可全部离线验证
7B 云模型接入（OpenHarness 或内置 loop，DashScope 首选）          —— 需要 key；含 .agent_config 加载与脱敏日志
7C 前端会话 UI + action 执行回路                                 —— 与 7B 可并行
7D 端到端验收（婚礼场景脚本）
```

## 7. 验收标准

- [ ] mock 模型集成测试离线全绿：脚本化工具调用序列（face_state → recommend → search → apply）逐事件断言。
- [ ] 真实场景："下周参加婚礼，帮我挑一支显气色的口红，先让我看看效果" —— agent 自主完成 ≥2 次工具调用，镜中上妆，并解释选择理由。
- [ ] 多轮指代："换个淡一点的" 在同会话中正确换成更浅的候选。
- [ ] 简单问题（"我适合什么妆"）实测仍走原路径，延迟与阶段 3 持平。
- [ ] 断网/无 key 时 agent 入口优雅降级，主链路零影响。
- [ ] 隐私：抓包/日志确认发往云端的请求体不含任何图像数据；API key 不入 git。
- [ ] 路由与工具层单测 + 现有 18+ 测试回归全绿；日志 T 系列记录；提交。

## 8. 风险

- 多轮工具调用 × 云端延迟（每轮 1–3s）→ 全程可能 5–10s：用 status 事件流式展示进度缓解，工具轮数上限 6。
- 小概率模型输出畸形工具调用：协议层校验 + 一次重试 + 失败降级普通回答。
- OpenHarness 接入是最大未知数，已设预算上限（1 次会话）与替代路径。
- API key 计费：默认 qwen-plus（便宜一个数量级），prompt 内限制回答长度。
