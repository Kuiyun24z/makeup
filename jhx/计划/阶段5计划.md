# 阶段 5 计划：魔镜语音交互（语音输入 → 指令试妆 → 唤醒词）

> 2026-06-11 制定。本计划同时回答"语音输入和美颜升级哪个优先"，并把阶段 5 按魔镜产品愿景重新设计。只规划，不施工。

## 0. 优先级结论：语音链路优先，美颜升级押后

理由：

1. **"帮我试试 XX 口红"的瓶颈不在渲染，在指令链路。** 现有 canvas 口红/腮红渲染已经能"上妆"，差的只是好看程度；但"听懂 XX 口红并自动换色"这条链路（识别→解析→色库→应用）目前完全不存在。先做缺失的链路，后升级已有的渲染。
2. **语音+魔镜是产品差异化主线**，美颜是所有美妆 app 的同质能力。Demo 给人看时，"对着镜子说话它就换妆"远比"磨皮更细腻"有冲击力。
3. **美颜升级（face parsing 精确唇形、原生引擎）工程量大、依赖多**，阶段 6 已有规划。且指令链路做好后，升级美颜只是替换渲染后端，`makeupSettings` 接口不变，两者解耦。

顺序：**5A 指令→试妆（文字驱动先行）→ 5B 语音输入 → 5C 唤醒词 →（5D 可选语音回复）**，美颜升级保持在阶段 6。

## 1. 5A：指令解析 + 口红色库（链路的消费端，先用文字验证）

核心洞察：语音只是输入通道，"试试豆沙口红"打字进来也应该能换妆。先用文字把整条链路跑通，语音接上即用。

**改动：**

- 新建 `web_demo/cosmetics.json` 本地色库：
  - 每条：`{id, name, aliases[], category(lip/blush), rgb, finish, defaultIntensity}`；
  - 第一版 15–25 条：通用色系（正红/豆沙/枫叶红/蜜桃/浆果/奶茶…）+ 少量俗称别名（"姨妈色"→深浆果 等）；
  - 不绑定真实品牌色号（商品库是阶段 6 的事），但 schema 预留 `brand`/`sku` 字段。
- 前端新增指令解析器（纯规则，不依赖 LLM）：
  - 模式：`试(试|一下)?\s*(.+?)(色)?(的)?(口红|唇釉|腮红)`、`换成…`、`卸妆/恢复`、`口红深一点/浅一点`；
  - 色名匹配：先精确匹配 name/aliases，再做包含匹配；命中 → 更新 `makeupSettings`（颜色+强度）并即时渲染；
  - 未命中色库 → 把问题转给现有问答流（顾问手记照常回答），不让指令"失败"。
- 提问框升级为"指令或问题"统一入口：先过指令解析器，不是指令再走 `/api/advice/stream`。
- 处方卡片联动：换色后"试妆工具"卡显示当前色名。

**验收：** 输入"试试枫叶红口红"≤0.5 秒换色；"口红深一点"调强度；"卸妆"复原；"我适合什么妆"仍走顾问问答；色库未收录的色名得到友好回应而非沉默。

## 2. 5B：语音输入（按住说话）

**方案选型（现实约束要先说清楚）：**

| 方案 | 优点 | 风险 |
|---|---|---|
| 浏览器 Web Speech API | 零依赖、代码量最小 | Chrome 走 Google 服务器、Edge 走 Azure——**联网依赖，国内网络环境可能不可用**；离线必挂 |
| Vosk WASM（本地） | 完全离线、隐私好，符合本项目"全本地"基调 | 中文小模型 ~42MB 需 vendor；集成工作量中等 |
| 后端 faster-whisper | 准确率最高 | 装依赖重、首字延迟高，按住说话场景体验一般 |

**决策：双路径，自动探测。** 首选 Web Speech API（Edge/Chrome `zh-CN`，代码 ~50 行），启动时探测可用性；不可用或识别失败 ≥2 次自动切换/提示下载 Vosk 本地模型（放 `vendor/vosk/`，复用 MediaPipe 的 vendor 化套路）。后端 whisper 不做。

**交互：** 提问框旁加麦克风按钮，按住说话/松开提交（桌面也支持点击开始-点击结束）；识别中实时显示中间结果；识别文本直接进 5A 的统一入口。麦克风权限仅在首次按下时申请。

**验收：** 说"试试豆沙色口红"→ 松手 ≤1.5 秒内换妆；说"我适合什么妆"→ 进入顾问问答；断网时（若走 Vosk 路径）功能不受影响。

## 3. 5C：唤醒词"魔镜魔镜"

**方案选型：**

| 方案 | 评估 |
|---|---|
| Web Speech 持续识别匹配关键词 | 实现快，但持续联网识别、自动重启循环不稳定，只能当原型 |
| Vosk WASM 持续小模型 + 关键词匹配 | 全本地、可行性最好；partial result 里匹配"魔镜"即唤醒 |
| Picovoice Porcupine 定制唤醒词 | 专业方案但商业授权 + 定制中文唤醒词需其控制台，暂不引入 |

**决策：** 基于 5B 的 Vosk 路径做（这也是 5B 选 Vosk 兜底的额外理由）。唤醒后：播放提示音 + 自动展开建议区/弹出对话态 + 进入 8 秒聆听窗口，窗口内的话直接进统一入口；超时回到待机。

**硬性约束（隐私）：**

- 常开麦克风**默认关闭**，UI 上有明确的"魔镜待机"开关和呼吸灯指示；
- 所有唤醒检测在本地完成，待机期间不向任何服务发送音频；
- 计划 §8 隐私原则补充"常开麦克风需用户显式开启"条目。

**验收：** 开启待机后，1.5 米内正常音量说"魔镜魔镜"，≥8/10 次成功唤醒；电视/音乐背景下误唤醒 <1 次/小时（粗测）；关闭开关后确认无音频处理。

## 4. 5D（可选彩蛋）：魔镜开口说话

- 用浏览器 SpeechSynthesis（Windows 自带中文语音），把顾问手记第一句朗读出来；零依赖，~20 行。
- 唤醒成功时回应"我在"。
- 不满意系统音色就砍掉，不投入更多。

## 5. 实施顺序与工作量粗估

```text
5A 指令解析 + 色库      —— 1 次会话可完成（纯前端 + JSON）
5B 语音输入双路径       —— Web Speech 半天等级；Vosk vendor 化另算一次会话
5C 唤醒词              —— 依赖 5B 的 Vosk 路径，单独一次会话 + 实测调阈值
5D TTS                 —— 顺手
```

每个子阶段独立验收、独立提交，5A 完成后产品就已经支持"打字试口红"，语音只是让它"免手"。

## 6. 与美颜升级（阶段 6）的接口约定

- 指令链路只写 `makeupSettings`（颜色/强度/开关），不关心渲染怎么画；
- 阶段 6 升级渲染（face parsing 精确唇形 mask、更真实的混合模式、或原生引擎）时，保持 `makeupSettings` schema 不变，链路零改动；
- 色库 `cosmetics.json` 届时扩展为商品库（加品牌/色号/价格/链接），id 不变。

## 7. 风险清单

- Web Speech API 的联网依赖是最大不确定项，**5B 开工前先花 10 分钟在目标机器的 Edge/Chrome 上实测** `zh-CN` 可用性，结果决定 Vosk 的优先级。
- Vosk 中文小模型识别"枫叶红"这类专有色名准确率未知，色名匹配要做拼音级模糊容错的预案。
- 唤醒词两个字重复（魔镜魔镜）对小模型反而友好，但阈值需实测调整。
- 浏览器要求麦克风/持续音频必须 HTTPS 或 localhost——当前 localhost 部署没问题，将来对外部署需 HTTPS。
