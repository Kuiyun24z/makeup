# 阶段 2 计划：规则引擎（正确性层）

> 2026-06-11 制定。对应总计划"阶段 2"。目标一句话：不调用任何 LLM，也能从 FaceState 生成一份结构化、可信、可测试的美妆建议草稿。LLM 之后只负责把这份草稿说得好听（阶段 3）。

## 1. 范围

**做：**

- 新增 `web_demo/rules.py`：纯函数 `generate_draft(face_state: dict) -> dict`，无 I/O、无随机、无网络。
- 新增 `POST /api/draft` 接口：直接返回草稿 JSON，不经过 Ollama。
- 改造 `/api/advice`：先生成草稿，prompt 改为"润色以下结论，不得更改结论本身"；现有 `fallback_advice()` 替换为草稿的文字化渲染。
- 新增 `web_demo/tests/test_rules.py` 单元测试 + 固定 FaceState 测试样例。

**不做（避免范围蔓延）：**

- 不接商品库、不推荐具体品牌色号（阶段 6）。
- 不改流式输出（阶段 3）。
- 不动前端 UI，前端最多加一个"仅规则建议"的调试入口（可选）。

## 2. 输入

输入就是前端现在已经在发的 `pruneAnalysisForAdvice` 载荷，关键字段：

- `faceDetected` / `lighting` / `clarity` / `framing` / `stable`
- `pose`：`{ok, label, yaw, rollDeg, pitchRatio}`
- `featureSummary.gated`
- `featureSummary.colors`：`{confidence, skin: {undertone, depth, warmth, luma, hex}, lip: {family, depth, hue, saturation, value, hex}}`
- `featureSummary.face / eyes / lips`：脸型、眼距、眉眼距、唇部比例标签与数值
- `makeupPreview`：当前试妆设置

缺字段必须容错：任何字段缺失时输出降级草稿，不抛异常。

## 3. 输出 Schema（草稿 JSON，schemaVersion: 1）

```json
{
  "schemaVersion": 1,
  "quality": {
    "ok": true,
    "issues": [],
    "retakeHint": null
  },
  "confidence": { "colors": "normal", "geometry": "normal" },
  "base": {
    "undertone": "warm",
    "tone": "偏黄暖调底妆",
    "recommendFamilies": ["珊瑚", "蜜桃", "砖红", "豆沙"],
    "avoidFamilies": ["冷紫调粉", "蓝调玫红"],
    "note": null
  },
  "lips": {
    "recommendFamilies": ["豆沙", "蜜桃奶茶"],
    "saturation": "medium",
    "prep": null
  },
  "eyesBrows": {
    "browHint": "保持自然眉峰",
    "techniques": ["眼尾拉长", "浅色眼影减压"]
  },
  "contour": {
    "blushPlacement": "横向打腮红",
    "contourHint": "轻修容拉长"
  },
  "focus": ["横向腮红", "眼尾拉长"],
  "safetyNotes": []
}
```

约定：

- `quality.ok == false` 时，`base/lips/eyesBrows/contour` 全部为 `null`，只给 `retakeHint`。
- `issues` 枚举：`no_face` / `lighting_dark` / `lighting_bright` / `too_far` / `too_close` / `pose_gated` / `blurry`。
- `confidence.colors == "low"` 时，`base.note` 和 `lips.prep` 必须带"光线/角度影响，建议重测"说明。
- 输出只含妆容用语，禁止出现肤质病理、身份、年龄、性别推断词汇。

## 4. 规则表（第一版）

### 4.1 肤色 → 底妆与口红色系

| undertone | depth | 底妆 tone | 推荐色系 | 避免色系 |
|---|---|---|---|---|
| warm | 浅/自然 | 偏黄暖调 | 珊瑚、蜜桃、砖红、豆沙、暖橘 | 冷紫调粉、蓝调玫红 |
| warm | 小麦/深 | 暖调加深一阶 | 砖红、陶土、深豆沙 | 浅粉荧光、香芋紫 |
| cool | 浅/自然 | 偏粉冷调 | 蓝粉、玫红、浆果、蓝调正红 | 橘调、暖棕 |
| cool | 小麦/深 | 冷调加深一阶 | 浆果、酒红、冷玫瑰 | 橘红、珊瑚 |
| neutral | 任意 | 中性自然调 | 豆沙、玫瑰豆沙、MLBB、奶杏 | 无硬性避雷，提示极端冷暖色谨慎 |

### 4.2 唇色 → 饱和度与打底

- 唇色 `depth == 偏深`：推荐 `saturation: high` 或顺色深色系；`prep: "唇色较深，浅色系需先唇部打底"`。
- 唇色 `depth == 偏浅`：任意饱和度；推荐先从 `medium` 开始。
- 唇色 family 与肤色推荐色系冲突时（如冷调肤色 + 橘调唇色），以肤色规则为准，并在 `lips.prep` 说明。

### 4.3 几何 → 眉眼与修容（沿用并接管现有 makeupFocus 逻辑）

- 眼距偏开 → 内眼角提亮；眼距偏近 → 眼尾拉长。
- 眉眼距离偏近 → 浅色眼影减压；偏开 → 眉下阴影衔接。
- 偏长椭圆 → 横向腮红；偏短圆润 → 轻修容拉长；下颌存在感强 → 下颌柔化。
- 唇部线条偏薄 → 唇峰和中部提亮。
- 规则引擎成为这些结论的唯一出处，前端 `makeupFocus` 后续从草稿读取（本阶段可暂保留前端副本，但以草稿为准）。

### 4.4 质量门（优先级最高，短路其余规则）

按顺序检查，命中即 `quality.ok = false`：

1. `faceDetected == false` → `no_face`，提示正对镜头。
2. `featureSummary.gated == true` → `pose_gated`，提示转正脸。
3. `lighting == dark|bright` → 对应 issue，提示调光线。**例外**：光线差但其余正常时，可输出几何类建议（眉眼/修容），仅色彩类置 null。
4. `framing == too_far|too_close` → 提示调整距离。

## 5. 集成点

- `server.py`：
  - `import rules`；`POST /api/draft` 返回 `rules.generate_draft(payload["analysis"])`。
  - `/api/advice` 的 prompt 结构改为：用户问题 + 草稿 JSON + 原始 FaceState（缩减版），并明确"你只能润色草稿结论，不能新增或推翻色系判断"。
  - Ollama 不可用时，返回 `render_draft_text(draft)`（草稿的简单中文拼接）代替现在的四句 `fallback_advice`。
- 前端：本阶段不强制改动。

## 6. 测试

`web_demo/tests/test_rules.py`，标准库 `unittest`，固定样例直接写在测试文件里（≥6 个）：

1. 暖调偏浅 + 正常质量 → 推荐含珊瑚/豆沙，避免含冷紫调。
2. 冷调偏深 + 正常质量 → 推荐含浆果/酒红，避免含橘调。
3. 中性调 → 推荐含豆沙/MLBB，avoidFamilies 为空或仅提示。
4. `gated == true` → quality.ok false，issue `pose_gated`，base 为 null。
5. `lighting == dark` → 色彩字段 null，几何建议保留。
6. `faceDetected == false` → 仅 retakeHint。
7. 唇色偏深 → saturation high 且 prep 非空。
8. colors.confidence low → base.note 非空。
9. 字段缺失（空 dict 输入）→ 不抛异常，返回降级草稿。
10. 同一输入调用两次输出完全一致（确定性）。

运行方式：`python -m unittest discover web_demo/tests`（不需要 Ollama、不需要浏览器）。

## 8. 已知局限（写明，不算验收失败）

- 冷暖判定阈值是经验值，受白平衡影响；准确率提升依赖阶段 6 的 face parsing 区域均色与后续校准。
- 规则表第一版只覆盖色系/技法的粗粒度建议，不含质地（哑光/水光）细分，留到商品库阶段一起做。
