# 阶段 3 计划：LLM 渲染层与流式输出

> 2026-06-11 制定。对应总计划"阶段 3"。目标一句话：建议不再是 120 秒黑盒等待——结构化草稿卡片立即出现，模型润色文字流式逐字出现；渲染不再依赖模型输出格式。

## 1. 设计决策（先定调）

阶段 2 之后，建议的"结论"已经全部在草稿 JSON 里。所以本阶段不做"让模型输出 JSON 再解析"（小模型格式不稳，又回到正则的老路），而是：

- **结构化卡片直接由草稿渲染**：底妆/唇妆/眉眼/重点卡片来自 `draft`，请求发出后立刻显示（不等模型）。
- **模型输出降级为一张"顾问解读"卡**：纯自然语言，流式追加，坏了也不影响结构化卡片。
- 现有 `splitAdviceIntoCards` 正则切卡逻辑整体删除。

这样"2 秒内看到内容"由草稿卡天然保证，模型首 token 慢只影响解读卡。

## 2. 范围

**做：**

- 后端新增 `POST /api/advice/stream`（NDJSON 流式接口）。
- 保留现有 `POST /api/advice`（非流式）作为兜底和调试接口，行为不变。
- 前端改造建议区：草稿卡片渲染 + 流式解读卡 + 请求中断（AbortController）。
- 离线集成测试：用 stdlib 起一个 mock Ollama，全流程不依赖真模型。

**不做：**

- 健康轮询、rAF 试妆渲染、临时文件清理（阶段 4）。
- 语音（阶段 5）。qwen3 thinking 字段处理：仅当切换模型时再做，本阶段固定 gemma3:4b。

## 3. 流式协议（NDJSON）

`POST /api/advice/stream`，请求体与 `/api/advice` 相同：`{question, analysis}`。

响应 `Content-Type: application/x-ndjson`，每行一个 JSON 事件，顺序约定：

```text
{"type":"draft","draft":{...}}                          ← 必为第一行，立即发出
{"type":"delta","text":"今天"}                           ← 0..n 行，模型增量文本
{"type":"done","advice":"<全文>","model":"gemma3:4b"}    ← 正常结束，最后一行
{"type":"error","error":"...","fallback":"<草稿渲染文本>"} ← 异常结束（代替 done）
```

约定：

- `draft` 事件不等 Ollama 连接，先算先发，每行发出后立即 flush。
- Ollama 不可用/中途断开：发 `error` 事件，`fallback` 来自 `rules.render_draft_text(draft)`，连接正常关闭。
- 客户端断开（用户取消/换问题）：服务端捕获 `BrokenPipeError`/`ConnectionAbortedError`，停止读取上游并关闭 Ollama 连接，不留挂起线程。
- 上游参数沿用：`temperature 0.4`、`num_predict 420`、整体超时 120s。

## 4. 后端改动（server.py）

- `_handle_advice_stream`：
  1. 读请求 → `rules.generate_draft` → 写 `draft` 行并 flush；
  2. `urllib` 以 `stream: true` 调 Ollama `/api/generate`，逐行解析 NDJSON，每个 `response` 增量转发为 `delta` 行；
  3. 读到 `done: true` 后发 `done` 行（拼接全文）；
  4. 任何 URLError/超时 → `error` 行 + fallback。
- 抽出纯函数 `parse_ollama_line(line: bytes) -> dict | None`（供单测）。
- 响应头不带 Content-Length，靠连接关闭定界（HTTP/1.0 语义），每行 `wfile.flush()`。

## 5. 前端改动（app.js / styles.css）

- `requestAdvice` 改为调用 `/api/advice/stream`，用 `fetch` + `ReadableStream` 逐行解析 NDJSON。
- 渲染结构：
  - 收到 `draft` → 立即渲染卡片组：
    - `quality.ok == false`：只渲染一张"请先调整拍摄"卡（retakeHint）；
    - 否则渲染：底妆（tone + 推荐/避免色系）、唇妆（色系 + 饱和度 + prep）、眉眼（browHint + techniques）、修容腮红、重点（focus 标签）；低置信时显示 note。
  - 收到 `delta` → 追加到"顾问解读"卡，流式期间显示打字光标动画；
  - 收到 `error` → 解读卡显示 fallback 文本，标题标注"本地规则建议（模型不可用）"。
- 中断逻辑：流式期间"生成建议"按钮变为"停止生成"；新提问先 `AbortController.abort()` 旧请求，保证不串流。
- 删除 `splitAdviceIntoCards` / `normalizeAdviceTitle` 及相关正则。
- 流式失败（如网络层不支持）自动回退调用旧 `/api/advice` 一次。
- styles.css：新增草稿卡片小样式（色系标签、avoid 删除线或灰显、打字光标动画），不动整体布局。

## 8. 已知局限（写明，不算验收失败）

- gemma3:4b 在本机 CPU 上首 token 可能仍要几秒，流式只是消除"黑盒等待感"，不缩短总生成时间。
- ThreadingHTTPServer 每个流占一个线程，本地单用户场景够用；并发化留给真正部署时再说。
- 浏览器端 NDJSON 解析按行切分，假定每行 < 64KB（实际事件远小于此）。
