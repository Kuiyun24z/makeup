# 美妆 AI 智能体项目计划

> 修订说明（2026-06-11）：本版重写了总体架构和实施阶段。核心变化：建议的正确性由规则引擎保证，LLM 降级为"文案渲染层"；OpenHarness agent 从系统骨架改为后期"咨询师模式"的可选功能，阶段顺序相应调整；实时链路从设计上不依赖任何 LLM。项目状态记录统一移至 `PROJECT_LOG.md`，本文件只保留目标、架构与阶段。

## 1. 项目目标

把当前文件夹中的美妆/人脸工具与本地模型整合为一个面向美妆服务的网站系统。

最终形态：

- 用户通过网站打开摄像头，实时看到自己的面部特征分析结果；上传图片作为调试和降级入口。
- 系统基于摄像头视频流进行人脸检测、关键点识别、颜色采样、脸部区域分割等结构化分析。
- 规则引擎基于结构化分析生成建议结论；LLM 把结论渲染为自然、贴合用户问题的中文表达。
- 后续可扩展实时试妆、商品推荐、语音回复、个性化档案，以及开放式多轮咨询的 agent 模式。

产品表述使用"人脸检测与面部特征分析"，不使用"人脸识别"，避免身份识别带来的隐私和合规压力。

## 3. 本地模型与分工

- `gemma3:4b`
  - 角色：文案渲染模型。输入结构化分析 + 规则引擎结论 + 用户问题，输出中文建议。
  - 明确不承担：工具调用、规划、结论判断。它编不出错误结论，因为结论不是它出的。
- `qwen3:4b`
  - 备选渲染模型。thinking 字段干扰问题在 agent 需求消失后不再阻塞，按需再处理。

本地模型不做 tool calling。工具调用可靠性需要云端模型（Sonnet 级），仅在阶段 7 引入。

## 4. 总体架构

四层设计，LLM 是"文案"而不是"指挥官"：

```text
第 1 层  浏览器采集分析层（推模式，持续运行，不依赖 LLM）
  摄像头 → MediaPipe 关键点 → 几何特征 + 关键点区域颜色采样
  → FaceState JSON（每 ~200ms 更新，缓存最新值）
        |
第 2 层  规则引擎（正确性层，纯代码，可单测、可离线跑）
  FaceState → 结构化建议草稿
  （肤色冷暖→底妆/口红色系；比例→眉眼/修容重点；质量差→重拍提示）
        |
第 3 层  LLM 渲染层（表达层，流式输出）
  FaceState + 规则草稿 + 用户问题 → 自然中文建议
        |
第 4 层  Agent 咨询师层（后期可选，云端模型）
  仅服务开放式多轮请求；工具围绕缓存状态设计：
  get_face_state() / recommend_makeup() / search_products() / render_preview()
```

核心原则：

- 实时链路（第 1 层）永远不等待 LLM；分析是"推"模式，结果缓存，谁需要谁读。
- 建议的正确性来自规则引擎，LLM 只负责把结论说得自然、回应具体问法。
- 主流程调用顺序固定，写死在代码里；不让模型"决定"调什么工具。
- Agent 只在请求真正开放（多轮、跨工具、顺序不确定）时介入，并且有路由：一键建议走直通路径，不付 agent 延迟税。
- 姿态门控：头部偏转超阈值时，脸型/眼距等几何结论标记为"暂不判定"，不输出失真结论。

## 5. 模型策略

### 5.1 本地 MVP 阶段（阶段 1–6）

- 渲染模型：`gemma3:4b`（备选 `qwen3:4b`）。
- 视觉：浏览器 MediaPipe 关键点 + 关键点区域颜色采样；face parsing ONNX 作为增强。
- 无 API key、零成本，可在 16GB MateBook 14 上运行。
- 必须实现流式输出：4B 模型出 400+ token 很慢，阻塞式请求体验不可接受。

### 5.2 云端增强阶段（阶段 7 起）

- Agent 模型：Claude Sonnet / GPT-5 系列 / Gemini 2.5 Flash / Qwen。
- 视觉理解、ASR 按需接入。
- 云端只用于咨询师模式和质量增强，不改变第 1–3 层的本地架构。

## 6. 实施阶段

### 阶段 0：环境确认 —— 已完成

Ollama、模型、Git/GitHub、OpenHarness client 验证均已完成，详见 `PROJECT_LOG.md`。
遗留的 OpenHarness tools 兼容问题随架构调整移至阶段 7，不再阻塞当前工作。

### 阶段 1：FaceState 完整化 —— 进行中

已有：摄像头、关键点、亮度/清晰度/构图/稳定性、脸型/眉眼/唇部几何特征。

待做（按优先级）：

1. 颜色采样（当前最高优先级，小改动大收益）：
   - 在脸颊、额头、唇部关键点围成的区域直接采样像素，计算平均色。
   - 输出 `skin_tone_hint`（冷暖倾向）和 `lip_color_hint`，写入 FaceState。
   - 不等 face parsing ONNX；它是后续精度增强。
2. 姿态门控：
   - 用 2D 关键点估算粗略 yaw/pitch（或开启 MediaPipe transformation matrix）。
   - 超阈值时几何结论置为"姿态不正，暂不判定"。
   - `eyeTilt` 等像素值全部归一化，消除窗口尺寸依赖。
3. 上传图片降级入口（计划承诺过，尚未实现）。

验收：正对摄像头时 FaceState 包含可信的几何 + 颜色字段；侧脸时不输出失真脸型结论。

### 阶段 2：规则引擎（正确性层）

- 纯 Python/JS 函数：FaceState → 结构化建议草稿（JSON schema 固定）。
- 规则覆盖：肤色冷暖→底妆/口红色系；唇色深浅→饱和度；眉眼比例→眉形/眼妆；质量差→重拍提示。
- 可离线跑、带单测用例（给定 FaceState 断言输出）。
- 安全边界：不诊断皮肤病、不身份识别、不推断敏感属性。
- 验收：不调用任何 LLM 也能返回结构化基础建议。

### 阶段 3：LLM 渲染层与流式输出

- 后端接 Ollama stream，前端 SSE/分块渲染，替换现有 120 秒阻塞请求。
- prompt 输入：FaceState + 规则草稿 + 用户问题；模型职责限定为表达，不得推翻规则结论。
- 后端把建议组织为结构化 JSON 再渲染，替换现有依赖模型输出格式的正则切卡片。
- 验收：提问后 2 秒内开始逐字出现建议；模型格式漂移不影响渲染。

### 阶段 4：网站 MVP 收尾

- 健康检查改为周期轮询，Ollama 中途掉线 UI 能反映。
- 试妆 overlay 渲染改 `requestAnimationFrame`（最近一次关键点插值），分析保持低频，消除口红"漂移"。
- 文件清理与临时目录管理。
- 验收：浏览器中完成摄像头 → 实时特征 → 结构化结果 → 流式中文建议的完整闭环。

### 阶段 5：语音输入

- 前端先用 Web Speech API 占位，后端再接 Whisper 或云端 ASR。
- 验收：语音 → 文字 → 渲染层 → 建议。

### 阶段 6：妆效预览与产品推荐

- PixelFree 链路重做（现有快照桥接是临时 hack，不可演进）：
  - 原生 demo 改为处理后输出图片文件，API 读回并在网页内展示，用完即删；
  - 取消覆盖写 `IMG_2406.png`、二进制 patch exe、无管理弹窗的做法；
  - 若重做成本过高，先冻结此线，保留浏览器轻量预览。
- face parsing ONNX 接入，提升区域颜色精度。
- 建立本地商品库：品牌、品类、色号、价格区间、适用肤色/风格标签。
- 验收：网页内看到妆效结果图；建议能关联示例商品。

### 阶段 7：Agent 咨询师模式（可选，云端）

触发条件：出现真正开放式请求，如"下周参加婚礼，预算 300，从这几支口红里选一支并试妆"。

- OpenHarness 作为编排层，模型用云端（Sonnet 级）。
- 工具围绕缓存状态设计：
  - `get_face_state()`：读最新 FaceState 缓存，不重新分析。
  - `recommend_makeup()`：包规则引擎。
  - `search_products()`：查商品库。
  - `render_preview()`：调妆效渲染。
- 路由：简单问题走第 3 层直通路径；只有复杂多轮请求进 agent。
- 此时再解决 OpenHarness/Ollama tools 兼容问题（或直接用云端模型回避）。
- 验收：开放式请求中 agent 自主完成查库 → 推荐 → 试妆的多步编排。

## 7. 需要用户提供或确认的内容

本地 MVP（阶段 1–6 主体）不需要 API key。

阶段 7 及增强项需要确认：

- 云端 agent 模型选择与 API key。
- ASR 服务选择与 API key；是否需要 TTS。
- 是否需要真实商品推荐及产品库来源。
- 是否允许保存用户图片、语音和分析记录。
- 网站风格定位：专业彩妆顾问 / 新手日常淡妆 / 带货导购 / 试妆工具。

## 8. 隐私与合规原则

- 明示摄像头、图片、语音用途。
- 默认不保存用户图片；任何落盘的摄像头帧（含妆效渲染中间文件）必须用后即删。
- 如需保存，要求用户授权并提供删除机制。
- 不做身份识别、不推断敏感属性、不进行医疗诊断。
- 肤质、痘痘、色斑只做视觉表观建议，不做病理判断。
